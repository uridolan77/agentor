# v1.0 RC — repository and delivery status

This document summarizes what the repository is fit for as of **Phase 15 / PR75.7**, without overclaiming production readiness.

## What is RC-ready (for an integration / evaluation stack)

- **Deterministic runtime path**: runs, steps, tool calls, policy decisions, traces, manifests, and in-memory or EF-backed persistence are exercised by automated tests.
- **API surface** (`/health`, `/ready`, `/api/v1/*`, Athanor-shaped routes, review/governance) is covered by contract and API tests; build and CI (restore, build, test, migration list, evaluation slice, benchmark compile, harness + repo-clean checks, Docker image, TRX) enforce regressions.
- **Documentation** for v1.0–RC scope, redaction, performance baselines, architecture boundaries, and release/deferred items is present and versioned in-repo.

## What is not production-ready by default

- **No claim** of multi-tenant security, high availability, or full operational SLOs; those require environment-specific hardening not asserted here.
- **External systems** (real Athanor, real Conexus, live MCP servers) are adapter boundaries; correctness of remote contracts must be validated per deployment.
- **Deferred acceptance rows** (see `feature-list.json` and `v1.0-RC-DEFERRED-ITEMS.md`) remain explicitly **false** until named evidence exists.

## Remaining `passes: false` harness items (snapshot)

Typical themes:

- **Running-run public API tests** for evidence/candidate flows while a run is `Running` (fixture-driven API tests not yet wired for every path).
- **Full PolicyBundle / enterprise policy engine** beyond `PolicyProfileRules` — roadmap **v1.1+**.
- **Multi-step plan executor resume** beyond the single pending-tool path — roadmap **v1.1+**.

See the harness and deferred-items doc for exact IDs and evidence strings.

## Architectural boundaries (stable intent)

- **Agentor** coordinates execution, policy, and observation; it does not replace Athanor, Conexus, or a host product identity plane.
- **Athanor** remains canonical for knowledge state when integrated; **Conexus** remains the model gateway; **MCP and external-agent protocols** are **ports/adapters**, not core domain.
- **Frameworks** (Semantic Kernel, Agent Framework, etc.) stay behind adapters if introduced.

## Operational limitations

- Default configurations favor **determinism and testability** (e.g., in-memory queues, fake adapters). Production profiles require explicit configuration review.
- Audit/export redaction is policy- and catalog-driven; extend catalogs per jurisdiction.
