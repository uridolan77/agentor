# v1.0 release candidate — tagging and harness alignment

This document describes how to cut a **git tag** for Agentor **v1.0 RC** while keeping repository harness truth consistent.

## Preconditions

1. **`main`** (or your release branch) is green: `dotnet restore` / `dotnet build --no-restore` / `dotnet test --no-build` on `Agentor.sln`.
2. **`pwsh ./scripts/verify-harness.ps1 -ExpectedPhase 40 -ExpectedHarnessPass PR170`** passes after harness updates for the tag.
3. **`pwsh ./scripts/verify-repo-clean.ps1`** passes (UTF-8 text, harness shape, acceptance evidence).
4. Deferred harness rows: **`passes: false` count = 0** in `.agentor-harness/feature-list.json` (see `docs/RELEASE/v1.0-RC-DEFERRED-ITEMS.md`).

## Tag naming

Recommended annotated tag form:

```text
v1.0.0-rc.1
```

The tag label should match **`AgentorRuntime:Version`** in `src/Agentor.Api/appsettings.json` and the default in `AgentorRuntimeOptions` for that commit.

## Tag workflow (example)

```bash
git checkout main
git pull
# confirm harness phase 40 / harnessPass PR170 on disk
git tag -a v1.0.0-rc.1 -m "Agentor v1.0.0-rc.1 release candidate"
git push origin v1.0.0-rc.1
```

## After tagging

- Build and publish the container image from the **same commit** as the tag (see `Dockerfile` and `docs/deployment/production.md`).
- Run **`docs/RELEASE/v1.0-RC-VERIFICATION.md`** checks on a candidate environment before promoting the image broadly.
- If you discover a harness typo **after** the tag, fix forward on `main` and add a patch tag or a new RC tag; do not rewrite published tags.

## CI alignment

GitHub Actions **verify-harness** must use the same **ExpectedPhase** and **ExpectedHarnessPass** as `.agentor-harness/feature-list.json` on `main` at the tagged commit (see `.github/workflows/ci.yml`).
